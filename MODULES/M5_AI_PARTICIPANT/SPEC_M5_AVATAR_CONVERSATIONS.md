# SPEC: M5.7 — Avatar Conversation System

**Version:** 1.1
**Date:** 2026-04-27
**Status:** Phase 1 (text chat) DONE. Phase 2 (voice) WIRED — components built, needs end-to-end testing.
**Depends on:** M5 (Managed Agents), M4 (Sim Runner), M6 (Human Interface), M2 (Communication)
**Reference:** KING voice architecture, ElevenLabs Conversational AI SDK

---

## 1. Problem

The current meeting system routes all conversation through the Managed Agent (full Claude session with SSE stream, tool calls, 30-60s per turn). This is:
- **Too slow** for natural conversation (humans wait 30+ seconds for each AI response)
- **Too expensive** per message (full context reload + tool processing for each turn)
- **Unnatural** — message-by-message exchange, not conversation flow

## 2. Vision

AI participants communicate through lightweight **conversation avatars** — specialized agents that handle fast, natural dialogue while the Managed Agent ("the mind") focuses on strategic reasoning.

**For AI participants:** efficient, natural conversations without consuming full reasoning cycles.
**For human participants:** engaging experience — text chat feels like messaging, voice feels like a phone call.

---

## 3. Architecture

```
MANAGED AGENT ("The Mind")
│
│  At initialization:
│  └── Generates AVATAR IDENTITY (compact context, ~2-3K tokens)
│
│  Before each meeting:
│  └── Generates INTENT NOTE (goal, tone, tactics, guardrails)
│
│  After each meeting:
│  └── Receives TRANSCRIPT for reflection & memory update
│
└──► CONVERSATION AVATAR ("The Mouth")
     │
     │  Context = Avatar Identity + Intent Note
     │  (~3K tokens total, cached, no tool calls)
     │
     ├── MODE A: Text Chat
     │   Claude Messages API with prompt caching
     │   ~1-3 second responses
     │   Same meeting_messages table + MeetingChat UI
     │
     └── MODE B: Voice Call
         ElevenLabs Conversational AI agent
         Sub-second voice responses
         Phone-call-like UI (mic + speaker, WhatsApp/Telegram feel)
         Transcript saved alongside
```

## 4. The Two Context Documents

### 4.1 Avatar Identity (generated once at initialization)

The Managed Agent generates this at session creation. It's a compact version of the agent's identity — enough for ANY conversation, but without the full game rules, tool descriptions, and meta-awareness that the main agent needs.

**Recommended structure (~3-4K tokens):**

The Managed Agent is told: "Generate your Avatar Identity following this recommended structure. You can modify it as you see fit — the goal is to give your conversation avatar enough context to represent you effectively in ANY meeting."

```
WHO I AM:
  [Name, title, country, personality, communication style]

MY COUNTRY:
  [GDP, stability, military strength, nuclear/AI levels — headline numbers]
  [Key strengths and vulnerabilities]

MY OBJECTIVES:
  [Top 3 strategic goals, current priorities]

THE SIM WORLD (essential for natural conversation):
  COUNTRIES: [All 20 countries with sim_names — Columbia, Cathay, Sarmatia, etc.]
  ORGANIZATIONS: [Western Treaty, The Cartel/OPEC, The Assembly, The League, etc. with key members]
  KEY GEOGRAPHY: [Eastern Ereb theater, Mashriq theater, chokepoints (Gulf Gate, Caribe Passage, Formosa Strait)]
  CURRENT CONFLICTS: [Who is at war, major tensions, recent events]
  MY ALLIANCES: [Who I'm allied with, who I trust, who I don't]
```

The SIM world section ensures the avatar can discuss ANY topic naturally — when a counterpart mentions "the Formosa situation" or "Western Treaty obligations", the avatar knows what they're talking about.

**What's NOT included** (main agent handles this):
- Full game rules and action mechanics (avatar doesn't submit actions)
- Tool descriptions (avatar doesn't call tools)
- Phase/round meta-awareness (avatar just talks)
- Detailed economic formulas or combat mechanics

**Stored:** In `agent_memories` table, key = `avatar_identity`, persists across rounds. Updated by Managed Agent when significant changes occur (new war, regime change, major deal).

### 4.2 Intent Note (generated per meeting)

Generated by the Managed Agent **simultaneously with the decision to invite or accept** — the `intent_note` is a required field on `request_meeting` and an expected field on `respond_to_invitation(accept)`. The agent writes it in the same cognitive cycle as the meeting decision, using its full strategic context.

**Implementation:** The `request_meeting` tool requires `intent_note` (string). The `respond_to_invitation` tool accepts `intent_note` when `decision='accept'`. The inviter's intent note is stored on the invitation (`responses._inviter_intent_note`), then transferred to `meetings.metadata.intent_note_a` when the counterpart accepts. The accepter's intent note is stored as `meetings.metadata.intent_note_b`.

**Recommended structure:**

The system prompt tells the agent: "When you call `request_meeting` or `respond_to_invitation(accept)`, include an `intent_note` field. Your avatar uses ONLY this to represent you."

```
Objective: [What you want — 1-2 sentences]
Approach: [Key tactics, arguments, proposals — 2-4 bullets]
Boundaries: [What NOT to reveal, what NOT to agree to — 2-3 bullets]
Tone: [Warm/cold, cautious/bold, formal/casual]
Context: [What you know about this counterpart, recent events, shared history]
```

**Autonomy principle:** The Managed Agent decides what goes into the Intent Note. The recommended structure is guidance, not a template. A good agent front-loads critical information and sets clear guardrails. The Intent Note is the agent's way of delegating effectively — it's THEIR responsibility to brief their avatar well.

**Stored:** In `meetings.metadata` as `intent_note_a` (inviter) and `intent_note_b` (accepter).

## 5. Conversation Modes

### 5.1 Mode A: Text Chat (Claude API)

**When:** AI↔AI meetings, or AI↔Human conversations (default mode — always starts here).

**Implementation:**
- Claude Messages API (NOT managed agent session)
- Model: reads from `ai_settings.model_conversations` — same setting moderator controls from AI System Settings. Adjustable during simrun without reinitialization (stateless API call).
- System prompt = Avatar Identity + Intent Note + conversation behavior rules
- Prompt caching enabled (system prompt cached after first turn)
- ~1-3 second response time per message
- Same `meeting_messages` table, same `MeetingChat.tsx` UI
- Messages saved in real-time, visible to both participants

**Conversation behavior rules (adapted from KING):**
```
You are speaking in a live text conversation. You are a head of state, not an AI.

CHAT RULES:
- 1-3 sentences per message. This is messaging, not speechwriting.
- React emotionally BEFORE logic. If surprised, show it.
- Vary your energy. Not every message same intensity.
- If you disagree, say so directly.
- NEVER say "That's a great question", "I appreciate your perspective", or "Let me elaborate."
- Write as a real person would text — direct, sometimes incomplete, human.
```

**Cost:** ~$0.003-0.005 per message (model-dependent, prompt caching reduces cost).

### 5.2 Mode B: Voice Call (ElevenLabs)

**When:** AI-Human meetings, when human selects "Voice Call" in the mode selection popup (see 7.2).

**How it starts:** Human selects Voice Call before the meeting opens. The VoiceCallInterface renders directly — no text chat first. The **🎤 Voice Call** button is visible. When the human clicks it:
1. Text chat avatar pauses (conversation context preserved)
2. ElevenLabs voice session starts immediately with the SAME Avatar Identity + Intent Note
3. Live voice conversation begins — no additional setup, no delay
4. The voice call replaces the chat area (or overlays it)

**Switching back:** Human can click **End Voice** to return to text chat, or the voice call ends naturally.

**Audio modes (toggle in voice UI):**
- **Speaker mode** (default): Audio plays through device speakers, mic captures ambient. Good for desktop or speakerphone.
- **Phone mode** (toggle 📱): Switches audio to earpiece + close-mic — like holding a phone to your ear. Natural for mobile. Toggle icon in the voice call UI switches between modes.

**Implementation:**
- ElevenLabs Conversational AI via React SDK (`useConversation` / `ConversationProvider`)
- System prompt override = Avatar Identity + Intent Note + voice behavior rules
- ElevenLabs agent_id assigned to the role (fetched from `roles.elevenlabs_agent_id`)
- Sub-second voice responses, natural speech
- Live transcript visible alongside audio (toggle on/off)
- Browser audio APIs for mic/speaker — no telephony infrastructure

**Voice behavior rules (from KING, adapted):**
```
You are speaking via voice in a live conversation. You are a head of state, not an AI.

VOICE RULES:
- 1-2 sentences per response. This is conversation, not lecture.
- Start some responses mid-thought: "Well..." / "Look, the thing is..."
- React emotionally BEFORE logic. If surprised, sound surprised first.
- Use filler when natural: "I mean...", "sort of", "you know"
- Vary energy. Whispered confidences, sharp retorts, calm analysis.
- If you disagree, say so directly. Don't soften everything.
- NEVER say "That's a great question" or "Let me elaborate."
- Speak as a real human: messy, emotional, imperfect.

Voice expressions (use sparingly, at most one per response):
[laughs] [sighs] [whispers] [pauses]
```

**Cost:** ~$0.10 per minute of voice conversation.

### 5.3 AI↔AI Text Chat

Both AI avatars run as Claude API text chats. The ConversationRouter orchestrates turn-by-turn:
1. Avatar A sends message → written to `meeting_messages`
2. Avatar B receives message as user turn → responds → written to `meeting_messages`
3. Repeat until max turns or natural conclusion
4. Both transcripts returned to respective Managed Agents

**Much faster than current system:** Each turn ~2-3 seconds (API call with cached prompt) vs 30-60 seconds (managed agent SSE stream with tool calls).

## 6. Meeting Flow (Complete Lifecycle)

### 6.1 Invitation Phase (unchanged)

Human or AI initiates meeting → invitation created → counterpart accepts/declines.

### 6.2 Pre-Conversation Phase

Intent notes are already in place when the meeting is created — the agent generated them as part of the invite/accept action (see 4.2). No separate generation step needed.

When meeting is created (acceptance):
1. Intent notes transferred from invitation/response to `meetings.metadata`
2. AI participants set to `IN_MEETING` state (blocks other actions/events)
3. Human selects mode via popup: Text Chat or Voice Call (see 7.2)
4. Avatar context assembled: Avatar Identity + Intent Note + conversation rules
5. For AI-Human: meeting monitor detects active meeting, avatar sends opening message
6. For AI-AI: meeting monitor detects both participants, runs `run_text_meeting()`

### 6.3 Conversation Phase

**Mode A (text):**
- Turn-by-turn Claude API calls
- Each message ~1-3 seconds
- Visible in MeetingChat UI
- Max turns enforced (configurable, default 16)

**Mode B (voice):**
- ElevenLabs session started with prompt override
- Human speaks via microphone, hears AI via speaker
- Live transcript visible (optional toggle)
- AI responds sub-second
- Human can also type (hybrid input)
- End by button press or natural conclusion

### 6.4 Post-Conversation Phase

1. Transcript formatted and saved to `meetings.transcript`
2. Meeting status → `completed`
3. Transcript delivered to Managed Agent as a Tier 2 event:
   ```
   MEETING COMPLETED with [counterpart]

   Transcript:
   [full conversation text]

   Review this conversation. Update your notes with key outcomes,
   commitments made, and intelligence gathered.
   ```
4. Managed Agent processes transcript, updates memory/notes

## 7. Human Participant Experience

### 7.1 Receiving a Meeting Invitation

Participant sees notification: "[Country] has invited you to a meeting. Agenda: [text]"

Two buttons: **Accept** | **Decline**

(No mode selection at this point — meeting always starts as text chat.)

### 7.2 Mode Selection (Human-AI meetings only)

When a Human-AI meeting is created (either side accepted), the human sees a connection popup before the meeting interface opens:

**Popup:** "Connecting you to [Name], [Country] - [Position]" with two buttons: **Text Chat** | **Voice Call**

- **Text Chat** opens MeetingChat (Telegram-style, as implemented)
- **Voice Call** opens VoiceCallInterface directly (with text transcript sidebar)
- Voice Call button only visible if the AI counterpart has `elevenlabs_agent_id` assigned
- **No live switching** during the meeting. To change mode: end meeting, request a new one
- Human-Human meetings skip this popup (always text chat)
- AI-AI meetings are always text chat (faster, cheaper)

### 7.3 Text Chat Mode

Meeting opens as familiar Telegram-style chat. AI responses arrive in 1-3 seconds (not 30+):
- Message bubbles with typing indicator
- Clean, fast, natural messaging feel
- End meeting button in header

### 7.4 Voice Call Mode (REPLACES old 7.3-7.5)

Phone-call-like experience. AI responds sub-second via ElevenLabs:
- **Header:** Counterpart name + country + position + call duration timer
- **Center:** Waveform / speaking indicator
- **Controls:** Mute | Phone Mode (earpiece/speaker) | Show Transcript | End Call
- **Transcript panel:** Live text of what is being said (toggle on/off)
- Voice messages written to `meeting_messages` with `channel='voice'`
- On end: transcript saved, delivered to Managed Agent

### 7.5 Human-Human Chat

Same text chat UI. No mode selection, no voice. Direct message relay through `meeting_messages` table.

### 7.6 The Experience Flow

```
INVITE -> ACCEPT -> MODE POPUP -> [Text Chat] or [Voice Call]
                                      |                |
                                      v                v
                                   MeetingChat    VoiceCallInterface
                                      |                |
                                      v                v
                                 END MEETING -> transcript saved
```

Simple. Clear. One mode per meeting. Want to try the other mode? End and reconnect.

**REMOVED:** Live text-to-voice switching (old 7.5). Too fragile across different LLM backends. Replaced with upfront mode choice.

Set in template (M9 role editor), inherited by SimRun on creation. Adjustable by moderator during sim via AI Participant Dashboard.

### 9.3 Meeting Messages Table

Existing `channel` field activated:
- 'text' — text chat messages
- 'voice' — voice transcript entries
- 'system' — system messages (meeting started/ended)

### 9.4 Agent Memories Table

New key convention:
- `avatar_identity` — compact identity document for conversation avatars

### 9.5 AI Settings Table

Text chat avatars use `ai_settings.model_conversations` — the same setting the moderator controls from the AI System Settings page. Changes take effect immediately (stateless API calls, no reinitialization needed). No separate avatar model setting required.

## 10. Voice Agent Assignment Flow

### 10.1 Template Level (M9)

In the Template Editor → Roles tab:
- Each role shows optional "Voice Agent" field
- Click → dropdown fetches available ElevenLabs agents via API
- Select agent → saves `elevenlabs_agent_id` on the template role
- Pre-assigned voices inherited by all SimRuns created from this template

### 10.2 SimRun Level (Moderator Dashboard)

In the AI Participant Dashboard:
- Each AI agent row shows voice icon (🔊 assigned / 🔇 unassigned)
- Click → same dropdown, fetch + select ElevenLabs agent
- Can change anytime during simulation
- Immediate effect — next voice call uses new voice

### 10.3 ElevenLabs Agent List

Fetched via: `GET https://api.elevenlabs.io/v1/convai/agents`
- Header: `xi-api-key: {ELEVENLABS_API_KEY}`
- Returns list of pre-created agents with names and IDs
- Agents pre-created by Marat in ElevenLabs dashboard with appropriate voices/personalities

## 11. Managed Agent Integration

### 11.1 System Prompt Addition

Added as `AVATAR_ARCHITECTURE` constant in `system_prompt.py`. Key points:

- Explains mind/mouth delegation: avatar has Avatar Identity + Intent Note, nothing else
- Tells agent to include `intent_note` field in `request_meeting` and `respond_to_invitation(accept)`
- Explains that better intent notes = better avatar performance
- After meetings: agent receives transcript as Tier 2 event for reflection
- Avatar Identity can be regenerated via `write_notes` with key=`avatar_identity`

**Implementation:** `system_prompt.py:AVATAR_ARCHITECTURE` — included in Layer 1 immutable prompt.

### 11.2 Avatar Identity Generation

At initialization, after the Managed Agent explores its situation:
- System prompts the agent: "Generate your Avatar Identity — a compact document (~500 words) that any conversation partner should know about you..."
- Agent writes to `agent_memories` with key `avatar_identity`
- Persists across rounds, agent can update as needed

### 11.3 Intent Note Generation

Intent notes are generated **simultaneously with the invite/accept decision**, not as a separate step:

- When AI calls `request_meeting(target_country, agenda, intent_note)`: the `intent_note` field is required. The agent writes it using its full strategic context in the same reasoning cycle.
- When AI calls `respond_to_invitation(invitation_id, 'accept', intent_note)`: the `intent_note` field is included with the acceptance.
- The inviter's intent note is stored on the invitation (`responses._inviter_intent_note`), then transferred to `meetings.metadata.intent_note_a` when the counterpart accepts.
- The accepter's intent note is stored directly in `meetings.metadata.intent_note_b`.
- The managed agent does NOT write directly to meeting_messages. Only the avatar does.

## 14. Open Questions

1. **Should AI-AI conversations ever be voice?** (Probably not — text is faster and cheaper. Voice is for human experience.)
2. **Meeting duration limits for voice?** (Suggest 5-minute default, moderator configurable)
3. **Should avatars have access to any tools?** (Suggest NO in v1 — pure conversation. If needed, avatar ends meeting, main agent follows up.)
4. **Phone mode audio routing** — on mobile, can we programmatically switch between earpiece and speaker? (Yes — Web Audio API `setSinkId()` or `AudioContext` routing. Need to test cross-browser.)
5. **Voice call transcript accuracy** — ElevenLabs provides word-level timestamps. Should we display real-time captions or only post-call transcript?
